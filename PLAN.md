# macOS用 音声入力アプリ 実装指示書

目的
macOS上で動く、AquaVoiceやTypelessに近い体験の音声入力アプリを作成してください。
ただし今回の前提は以下で固定です。

* 音声認識は Mac 側の on-device 処理を使う
* Gemini API は文章整形だけに使う
* まずは Gemini 2.5 Flash-Lite を標準モデルにする
* 品質重視モードだけ Gemini 2.5 Flash を使えるようにする

重要な方針
このアプリは「音声をそのままGeminiへ送るアプリ」ではありません。
音声はMacローカルで文字起こしし、Geminiへ送るのは文字起こし後のテキストだけです。
APIコストを低く抑え、速度とプライバシーを優先してください。

---

## ゴール

ネイティブの macOS メニューバーアプリを作成してください。
基本機能は以下です。

1. グローバルショートカットで録音開始・停止
2. マイク音声を取得
3. Macの on-device 音声認識でリアルタイム文字起こし
4. 録音停止後、文字起こし結果を Gemini API に送って整形
5. 整形後のテキストを、現在カーソルがある入力欄へ挿入
6. 挿入が難しい場合は、クリップボード経由で貼り付けにフォールバック
7. メニューバーからモード切替ができる
8. APIキー、モデル、整形ルールを設定できる

## 技術方針

可能な限り macOS ネイティブで実装してください。
Electron は使わず、Swift / SwiftUI / AppKit を基本にしてください。
必要に応じて AppKit を併用して構いません。

### 推奨構成

* UI: SwiftUI
* メニューバー: MenuBarExtra または AppKit の status bar
* 音声入力: AVAudioEngine
* 音声認識: Apple の公開 Speech 系フレームワーク
* 他アプリ入力: Accessibility API
* フォールバック貼り付け: NSPasteboard + 擬似 paste
* 設定保存: UserDefaults または AppStorage
* ネットワーク: URLSession
* ログ: os.Logger
* 非同期処理: Swift Concurrency

---

## 制約

以下は必ず守ってください。

* 生の音声データを Gemini API に送らない
* 文字起こしテキストだけを Gemini API に送る
* APIキーはソースコードに直書きしない
* APIキーは環境変数または設定画面から保存する
* 依存ライブラリは最小限にする
* 外部課金サービスや認証は不要
* まずは単体ローカル動作を最優先にする
* まずは英語と日本語の混在入力に配慮する
* UIはシンプルでよいが、状態は分かりやすくする

## アプリの動作仕様

### 基本フロー

1. ユーザーがグローバルショートカットを押す
2. 録音開始
3. リアルタイムで文字起こしを表示
4. 再度ショートカットを押す、または押下終了で停止
5. 文字起こし結果を整形ルール付きで Gemini API に送る
6. 整形済みテキストを取得
7. 前面の入力欄へ挿入する
8. 失敗したらクリップボードに入れて貼り付ける
9. それも失敗したら、結果をポップオーバーに表示して手動コピー可能にする

---

## グローバルショートカット仕様

競合アプリを参考に、macOS上で違和感の少ない音声入力起動UXを実装してください。

### 要件

* デフォルトの起動キーは「Fn」を第一候補とする
* ただし macOS の既存ショートカットやユーザー環境と競合する場合があるため、必ず変更可能にする
* 初回セットアップ時に「推奨ショートカット」を提示する
* 基本操作は「押している間だけ録音し、キーを離すと確定・整形・入力」を第一候補とする
* 将来拡張として「Fnダブルプレスで別モード起動」を追加しやすい設計にする
* Fn単体が環境依存で不安定な場合に備えて、Option + Space などの代替ショートカットへ安全に切り替えられること

### 最低限必要な挙動

* hold-to-talk: キー押下中に録音、キーを離したら処理開始
* shortcut customization: 設定画面から変更可能
* conflict handling: 競合時に警告または代替候補を表示
* future extensibility: 将来、単押し・長押し・ダブルプレスで動作を分けられる構造にする

### 注意

* Fn固定に決め打ちしないこと
* 競合アプリの体験を参考にしつつ、macOS純正機能との衝突を避けること
* MVPでは「1つのメインショートカット」を確実に安定動作させることを優先する

---

## モード

最低限、以下の3モードを実装してください。

### 1. プレーン整形

* フィラー削除
* 句読点補完
* 軽い誤字修正
* 話し言葉を自然な文へ変換

### 2. 丁寧文

* プレーン整形に加えて、文体を丁寧語に整える

### 3. 高品質整形

* Gemini 2.5 Flash を使う
* 長文や少し複雑な文章に対応
* 通常モードより遅くてもよい

### 標準モデル

* 通常: Gemini 2.5 Flash-Lite
* 高品質モード: Gemini 2.5 Flash

## Gemini への依頼内容

Gemini は「要約」ではなく「整形」が中心です。
意味を勝手に変えないでください。
事実追加、強い意訳、内容圧縮は避けてください。

### 整形ルール

* 「えー」「あのー」「そのー」などのフィラーを削除
* 明らかな言い直しを整理
* 句読点を自然に補う
* 話し言葉を自然な書き言葉に近づける
* 意味は変えない
* 不明瞭な箇所は勝手に補完しない
* 箇条書き口調なら箇条書きのままでもよい
* ユーザーが短文で話した場合は短文のまま過剰整形しない

Gemini には、できれば構造化レスポンスで返させてください。
推奨の返却形式は以下です。

* cleaned_text
* raw_text
* notes

ただし MVP では `cleaned_text` のみでも構いません。
まずは安定動作を優先してください。

## Gemini プロンプト方針

システム指示として、以下の趣旨を含めてください。

* あなたの役割は音声入力補助の文章整形である
* 要約ではなく整形を行う
* 意味を変えない
* フィラー除去、句読点補完、軽微な誤字修正だけ行う
* 情報を足さない
* 不明な固有名詞は無理に修正しない
* 出力は整形済み本文のみ、または指定JSONのみ

## 他アプリへの入力

第一候補は Accessibility API を使った挿入です。

## 品質要件

* 録音開始から文字起こし開始までが遅すぎないこと
* 録音終了後の整形が数秒以内で返ることを目標にする
* API通信失敗時にアプリが固まらないこと
* 前面アプリへの挿入失敗時も結果を失わないこと
* 連続使用してもメモリリークしにくいこと
* 無音や短い発話でも破綻しないこと
